# 仪表盘通用 Skill

> 用途：沉淀**跨项目可复用**的仪表盘设计、性能、数据更新和交付规范。  
> 边界：这里只写可复用方法，不写某个仓库独有的文件名、业务口径或临时补丁。  
> 配套：项目事实放进项目级 Skill；每次稳定迭代后，两份 Skill 一起更新。

---

## 一、双 Skill 分工

### 1.1 项目 Skill 负责什么
记录**当前仓库专属事实**：
- 文件路径
- 页面结构
- 业务字段口径
- 当前用户偏好
- 当前版本已经确认的限制条件

### 1.2 通用 Skill 负责什么
记录**以后别的驾驶舱也能直接复用**的方法：
- 页面布局套路
- KPI 与图表的视觉规则
- 图表调度与性能优化方法
- 数据导入 / 导出设计
- 维护与验收 checklist

### 1.3 维护约定
每次对话里只要做了这几类事，就要同步回写 Skill：
- 改了稳定的布局规则
- 改了主题风格偏好
- 修了可复用的性能问题
- 确认了新的导入导出规范
- 明确了新的维护原则或禁区

---

## 二、通用布局规范

### 2.1 页面拆分原则
典型驾驶舱建议拆成：
1. 总览页
2. 进展 / 经营页
3. 专项分析页
4. 数据管理页

### 2.2 KPI 区原则
- 首页 KPI 优先放在首屏。
- KPI 数量宁可少而准，也不要重复堆数字。
- 重要大盘指标优先用**等宽栅格**，减少视觉重心漂移。
- KPI 数字色和风险语义保持一致：中性蓝 / 正向绿 / 风险黄红。

### 2.3 图表区原则
- 图表按阅读顺序排，不按实现顺序排。
- 第一屏优先放“解释大盘”的图，而不是细枝末节图。
- 同一模块里图表尽量保持统一高度和留白节奏。
- 能用摘要卡解决的信息，不要强行再上一张图。

## 三、通用视觉规范

### 3.1 深色主题
- 深色页面不是越黑越高级，重点是**层次和发光控制**。
- 更推荐四周均匀 halo，而不是角落热点或廉价霓虹边。
- 蓝色主色在深底上通常需要比设计稿再亮半档，否则容易发灰发闷。

### 3.2 KPI 数字
- 默认蓝色数字建议使用高亮青蓝 / 冰蓝渐变，深底上要主动再亮半档（参考值：`#86e0ff → #d6fbff`，glow 强度 0.55–0.65）。
- 数字发光要服务于可读性，不要让 glow 抢过文字本身。
- 同一卡片内：数字色、顶部高光线、边缘 halo 最好同色系联动。

### 3.3 禁止项
- 扫描线动画
- 无意义的数据来源/最后更新时间装饰字样
- 黑色硬边框
- 重复 KPI
- 只为“看起来复杂”而增加的装饰图层

## 四、图表性能通用规范

### 4.1 首屏优先原则
- **当前页优先，隐藏页后置。**
- 首屏不要先渲染隐藏页的大表格、大列表、次级图表。
- 页面切换时再补对应页内容，用户体感会明显更快。

### 4.2 图表初始化原则
推荐使用：
- 页面级任务表
- 可取消调度
- 脏页标记
- 实例复用
- 分帧渲染

推荐模式：
```js
scheduleInitChartsForPage(page, { delay, force })
DIRTY_CHART_PAGES
renderNextChart(tasks, index)
```

### 4.3 避坑清单
以下情况最容易把仪表盘做慢：
1. 切页时先手动渲染一轮，再懒加载重复渲染第二轮
2. 主题切换时把全部图表实例销毁后整页重建
3. 首屏初始化时人为加 300ms / 500ms 固定等待
4. 数据刷新时不区分当前页和隐藏页，所有模块一起重算
5. 筛选后直接全量重建，而不是更新受影响部分
6. `syncDashboardFromCurrentDB` 里无条件渲染所有页的表格/摘要卡，会在首屏把产线、数据管理等隐藏页的重模块一起算一遍；应该按当前页条件分支，隐藏页内容等进入时再触发

### 4.4 更稳的刷新策略
- 主题切换：标记脏页，仅重绘当前页，其他页等进入时再刷新。
- 数据刷新：只刷新当前页图表与可见数据模块。
- 筛选联动：优先更新表格、摘要和当前页图表，不要全局重刷。

## 五、数据导入导出通用规则

### 5.1 导入
- 优先支持 Excel 与 CSV 两种入口。
- 列头尽量兼容中英双语、旧版别名和人工维护模板。
- 导入后统一走一个同步入口刷新页面，避免每个模块各自补丁式更新。

### 5.2 导出
- 页面级导出和数据管理页导出要分层。
- 用户最常用的是当前页导出，不一定是全量导出。
- 筛选态下的导出要默认跟随筛选结果。

## 六、交互与维护原则

### 6.1 调整原则
- 用户如果只要求调位置，就优先只动位置。
- 不要顺手改一堆无关样式。
- 能小修复解决，就别大改结构。

### 6.2 验收原则
每次稳定迭代后至少检查：
- 页面切换是否正常
- 当前页图表是否只渲染必要内容
- 深浅主题视觉是否一致
- 表格 / 筛选 / 导出是否被带坏
- 首屏体感是否比之前更快，而不是“理论更优”

### 6.3 文档回写原则
一旦本次改动具有复用价值：
- 项目事实回写到项目 Skill
- 通用方法回写到本文件
- 不把临时实验、失败尝试、偶发报错写进 Skill

### 6.4 分享快照原则
- 默认交付主文件，不额外长期维护 share 克隆文件。
- 用户明确要外部分享时，再从主文件生成**带时间戳的一次性快照**上传。
- 快照文件和公开链接都视为临时交付物，不要再反向当成长期主版本源。

---

## 七、推荐检查清单

每次交付前至少过一遍：
- [ ] 页面结构没被顺手改乱
- [ ] 没新增重复 KPI
- [ ] 没引入固定等待导致首屏变慢
- [ ] 没出现图表重复渲染
- [ ] 当前页优先策略仍成立
- [ ] 用户这次明确的视觉偏好已经沉淀进 Skill
- [ ] 项目 Skill 与通用 Skill 都已同步
